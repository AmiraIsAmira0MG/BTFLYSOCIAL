Project Status & Overview

BTFLY is a decentralised, federated social media platform built around the principle that users — not corporations — should own their presence online. The project is being built by a solo developer and is nowhere near completion at this stage. A basic working model exists, but it has not yet been implemented in its intended form.

This document exists to capture the intended architecture clearly so that development, future collaborators, and any interested parties understand how the system is meant to work.

⚠ Development Status

The project is openly early-stage. A proof-of-concept exists but the full system described here — Global Login, node federation, server types, and monetisation — is yet to be properly built out. Expect this spec to evolve significantly.

02

Global Login & Identity

At the core of BTFLY is the Global Login — a centralised identity layer that sits above the federated node network. Every user has one Cloudlight account, which acts as their master identity across the entire platform.

This is intentionally distinct from the self-hosted node layer beneath it. The Global Login is managed centrally at login.btfly.social and serves two critical purposes: it enables a consistent identity across all nodes a user participates in, and it provides the infrastructure for platform-wide moderation.

🌐
One identity
A single Cloudlight account ties together all of a user's node presences under one global identity.
🔑
Node accounts
Each Global Login can have multiple node accounts — one per server — each with its own local profile and posts.
🛡️
Moderation anchor
Because every account traces back to a Cloudlight identity, platform-level bans are enforceable globally.
ℹ Cloudlight Tie-in

The Global Login is shared with the broader Cloudlight ecosystem. This is the same account system used across Cloudlight's other products, giving BTFLY a built-in user base foundation at launch.

03

Node Servers

The self-hosted layer of BTFLY is made up of node servers — independently operated instances that communities or individuals run themselves. This is the federated part of the platform.

Anyone wanting to run a node downloads the server software from GitHub, runs it on their own hardware or hosting, and their instance immediately becomes part of the BTFLY network (within the rules of its chosen server type).

Nodes operate under their own domain — for example, example.social — and users access them via that domain's login page.

# User flow to join a node
1. Download server from GitHub → run locally or on a VPS
2. Node becomes accessible at login.example.social
3. User visits login page → node generates a one-time key
4. User redirected to login.btfly.social/[key]
5. User authenticates with their Cloudlight Global account
6. User enters their node's address: example.social
7. BTFLY asks example.social: "Did this key originate here?"
8. Verified → user logs in or creates node account

This flow ensures that only legitimate login attempts originating from a specific node can complete authentication for that node — protecting both the user and the server operator from spoofing.

04

Server Types

All BTFLY nodes operate as one of three server types. The type determines how the node participates in the broader network — how visible it is, how it handles federation, and whether its content persists if the server goes offline.

Dark
Dark Server
A fully private, isolated node. Dark servers are not discoverable on the global network and have no contact with other servers by default. Accounts on a Dark server can only interact with other accounts on the same Dark server. This is designed for closed, private communities that want full isolation.

An optional toggle allows Dark server accounts to view posts from other networks in read-only mode — they can see but cannot interact, reply, or federate.
Global discovery Cross-server interaction Post replication Full isolation Optional read-only view
Grey
Grey Server
A participatory node that operates as a normal federated account on the broader BTFLY network. Grey servers can interact with accounts on other servers — following, replying, liking — just as any user would.

The key distinction is that Grey servers do not replicate posts to other servers. Content lives exclusively on the Grey server. If the server goes offline or shuts down permanently, its posts and accounts are lost — there is no external backup.
Global discovery Cross-server interaction Post replication Content persistence after shutdown Standard federation
Light
Light Server
The highest participation tier. Light servers replicate their posts across the network, meaning content persists even if the originating server shuts down. This is the intended default for public-facing communities who want their content to remain accessible long-term.

Light servers can also optionally operate as replication-only nodes — servers that don't host their own community but instead act as distributed storage for other Light servers, strengthening the network's resilience.
Global discovery Cross-server interaction Post replication Content persists after shutdown Optional: replication-only mode
ℹ Replication nodes

A Light server in replication-only mode hosts no users and runs no community — it exists solely to mirror and preserve content from other Light servers. This allows volunteer operators to contribute to network resilience without running a full social instance.

05

Authentication Flow

The login process bridges the self-hosted node with the centralised Global Login in a way that's both secure and user-friendly. No password is ever sent to the node directly — authentication happens through BTFLY's central auth service.

Login sequence
🌐
User visits login.example.social
🔐
Node generates one-time key
🦋
Redirect to login.btfly.social/key
👤
User signs in with Cloudlight account
📍
User enters node address
✅
BTFLY verifies key with node
🏠
Access granted to node account

The key verification step — where BTFLY asks the node whether a given key originated there — prevents login hijacking. A key can only be used once and is only valid for the server that generated it, within a short time window.

06

Moderation & Global Bans

Because every node account is tied to a Cloudlight Global Login, the platform retains the ability to enforce global bans at the identity layer. If an account is found to be in violation of laws or platform-level rules — not just a node's community rules — the underlying Global Login can be banned, which propagates across all nodes simultaneously.

This is an intentional design choice. Fully decentralised networks struggle to deal with bad actors who simply migrate to new nodes after being banned. By anchoring identity globally, BTFLY can act decisively when legal violations or severe platform-wide abuse occurs, without relying on individual node operators to act.

📋 Moderation layers

Node-level: Server operators manage their own communities under their own rules. Platform-level: BTFLY can issue global bans via the Cloudlight identity system for legal violations or severe platform-wide abuse only. The intention is for platform-level intervention to be a last resort.

07

Monetisation

BTFLY's monetisation strategy is designed to be honest and transparent. The platform is openly early-stage and revenue isn't a current priority, but the intended model is as follows.

📣 Advertising

Ads will be introduced closer to a public release. This is acknowledged openly — the platform is not ad-free forever. Ads will only run on standard free accounts; Pro users will never see ads.

PRO
BTFLY Pro — Premium tier
Verification badge — a visible badge next to the user's profile icon across all nodes, similar to X's verified marker. Signals a paid, committed user.
No ads — Pro accounts have a completely ad-free experience across the platform.
Cloud backup of posts — posts made on any node are backed up centrally, so content isn't lost if a Grey server shuts down.
Cloud-hosted node add-on cost — for an additional fee, BTFLY hosts and manages the node server on the user's behalf, removing the need for self-hosting while retaining the federated model.

The cloud-hosted node add-on is separate from the base Pro subscription because the infrastructure cost is meaningfully higher. This keeps the base Pro tier affordable while offering a full managed option for those who want it.
